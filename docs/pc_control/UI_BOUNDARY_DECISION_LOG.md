# PCControl UI 边界重构决策记录

## 1. 记录原则

本文件记录执行过程中形成的设计决策、人工验证结论和后续重构待办。

禁止：

```text
把临时实现写成最终方案。
把未验证内容写成已验证。
把用户已否定的方案继续深化。
把不合理设计用兼容名义长期保留。
把“参考单设备设置”误写成“另起批量体系”。
```

---

## 2. 决策记录 2026-05-18：批量设置必须继承设备设置

### 2.1 用户纠偏

用户明确纠偏：

```text
批量设置要和设备设置一样，在设备设置的基础上改进。
```

准确理解：

```text
批量设置不是独立配置体系。
批量设置不是参考设备设置后另做一套页面。
批量设置本质上是 DeviceSettingsDialog 的批量模式。
批量设置应与单个设备设置保持同一套页签、同一套字段模型、同一套配置语义。
```

### 2.2 当前结论

当前 `ui/dialogs/batch_settings_dialog.py` 的第一轮独立页签设计先搁置。

该文件当前只能视为临时骨架和错误方向记录，不应继续扩展为最终方案。

当前独立页签方向包括：

```text
目标设备摘要
批量账号
批量任务
批量参数
差异预览
应用结果
```

这些不应作为最终批量设置主结构继续深化。

### 2.3 正确方向

批量设置应基于 `DeviceSettingsDialog` 演进。

当前单设备设置页签包括：

```text
主要设置
账号设置
任务设置
物品处理
购买设置
交易设置
制造设置
铸币设置
本机 ADB 连接
本地元数据
其他
```

批量设置应尽量保持同构：

```text
批量主要设置
批量账号设置
批量任务设置
批量物品处理
批量购买设置
批量交易设置
批量制造设置
批量铸币设置
批量本地元数据
批量其他
```

说明：

```text
本机 ADB 连接通常属于单设备本机连接映射，不适合直接批量覆盖。
如果后续需要批量处理 ADB 映射，必须单独设计，不应混入普通批量设置。
```

### 2.4 批量设置增加的不是新配置体系，而是覆盖策略

批量设置相比单设备设置，只增加批量维度能力：

```text
目标设备选择摘要。
字段级覆盖开关。
字段级覆盖策略。
批量差异预览。
敏感字段二次确认。
逐设备应用结果。
失败回滚或失败跳过记录。
```

每个字段应至少支持：

```text
不修改
覆盖为新值
清空
继承默认值
```

硬规则：

```text
未勾选字段不得覆盖。
密码等敏感字段默认不覆盖。
密码等敏感字段覆盖必须二次确认。
后端接口未联调前，不得声称批量应用成功。
```

### 2.5 建议后续实现方式

优先方向：

```text
抽取 DeviceSettingsDialog 的页签定义和字段定义。
让单设备设置和批量设置复用同一套字段模型。
批量设置只在字段旁增加“是否覆盖 / 覆盖策略”。
```

建议命名：

```text
ui/dialogs/batch_device_settings_dialog.py
```

或在重构时替换当前：

```text
ui/dialogs/batch_settings_dialog.py
```

但无论命名如何，最终结构必须表达：

```text
BatchDeviceSettingsDialog = DeviceSettingsDialog 的批量模式
```

而不是：

```text
BatchSettingsDialog = 一套独立的批量配置页面
```

## 4. 后续待办

### 4.1 P5 返工前置要求

后续返工 P5 时，不直接继续写批量页面代码。必须先完成：

```text
1. 精读 DeviceSettingsDialog 当前页签结构。
2. 梳理单设备设置字段模型。
3. 抽取可复用字段定义或页面构造规则。
4. 设计批量模式的覆盖策略。
5. 再实现 BatchDeviceSettingsDialog。
```

```text
P5 当前状态：已搁置。
原因：当前批量设置第一轮骨架没有做到与设备设置同构。
后续方向：基于 DeviceSettingsDialog 改进为批量设备设置模式。
```
